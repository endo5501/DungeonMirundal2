## Context

プロジェクトは `cursor-menu-ui` spec で「カーソルは固定幅列の `"▶"` を使い、独自プレフィックスは排除する」ことを合意済み。ほぼ全UIが `CursorMenu` + `CursorMenuRow` の組み合わせに移行しているが、`PartyFormation` (ギルドのパーティ編成画面) と、`DungeonEntrance` の一部表示ロジックは独自実装のままになっている。

`DungeonEntrance` はフロー設計面でも課題がある:
- 起動時にダンジョン一覧へフォーカスが当たる
- Enterでボタン列に焦点が移る
- ボタン列でもう一度Enterでアクション実行
- 結果として「対象選択 → 動詞選択 → 実行」の3段階モデル

Wizardry系RPGの自然な操作感は「動詞 → 対象 → 実行」であり、プレイ時に「まずダンジョンを選んでからでないと何もできない」という制約が直感に反している。

## Goals / Non-Goals

**Goals:**
- `PartyFormation` の待機中キャラクター一覧を他UIと同じ視覚表現 (`▶` 固定幅列) に統一する
- `PartyFormation` のパーティ格子 (inline 2D) もカーソル文字だけ `▶` に揃える
- `DungeonEntrance` のフローを「動詞 → 対象 → 実行」に反転する
- ダンジョン一覧はアクション選択中も情報として可視にする
- disabled 条件を新フローに合わせて整理する

**Non-Goals:**
- `PartyFormation` のパーティ格子を `CursorMenuRow` で完全に再構築すること (inline 2Dレイアウトは別物として割り切る)
- ダンジョン一覧とボタン列のレイアウト位置の大幅変更 (既存順序を尊重)
- `新規生成` ダイアログ・`破棄` 確認ダイアログの内部挙動変更
- 新しいアクションの追加

## Decisions

### 決定1: 待機リストは `CursorMenuRow`、格子は inline 置換のみ (B案)

**選択**: 待機リスト = 完全 `CursorMenuRow` 化、パーティ格子 = 定数 `"> "` → `"▶ "` に差し替えのみ

**代替案**:
- A: カーソル文字のみ差し替え (待機も格子もinlineのまま)
- C: 格子も各スロットを個別 `CursorMenuRow` 化

**理由**:
- 待機リストは1行1エントリの垂直メニューなので `CursorMenuRow` の「固定幅カーソル列 + テキスト列」と一対一対応する。ここを揃えると spec 準拠度が上がる
- パーティ格子は1行に3スロット並ぶ inline 構造で、`CursorMenuRow` の前提 (左端カーソル列) と合わない。個別スロット化は HBox 構成・Tab切替・フォーカスインデックス計算の再設計が必要で、UI一貫性という目的に対して労力が見合わない
- 格子部分は視覚的にも「メニュー行」というより「盤面」に近い性格で、cursor-menu-ui spec の「行レイアウト契約」が想定する対象ではない

### 決定2: `DungeonEntrance` は状態機械で「動詞先行」を表現

**選択**: 3つのモードを持つ状態機械:
```
BUTTON_FOCUS (default)
  ├─ [潜入] → LIST_FOCUS_ENTER
  ├─ [新規生成] → CREATE_DIALOG (既存)
  ├─ [破棄] → LIST_FOCUS_DELETE
  └─ [戻る] → back_requested

LIST_FOCUS_ENTER
  ├─ ↑↓: リストカーソル移動
  ├─ Enter: enter_dungeon 発火
  └─ ESC: BUTTON_FOCUS へ戻る

LIST_FOCUS_DELETE
  ├─ ↑↓: リストカーソル移動
  ├─ Enter: 破棄確認ダイアログ
  └─ ESC: BUTTON_FOCUS へ戻る
```

**代替案**:
- モーダル切替 (リストを選択中だけ表示): 情報量が減る
- 一覧とボタンを並置してTab切替: 現状と大差なく、結局「対象先行」のままになる

**理由**:
- 「動詞を選んだあとに対象を選ぶ」という宣言的な手順になり、認知負荷が下がる
- ダンジョン一覧は常に表示されるため、どのダンジョンが登録されているか事前に確認できる
- ESC でキャンセルしてボタン列に戻る経路が自然で、誤操作からの回復が容易

### 決定3: disabled 条件を「選択状態非依存」に再定義

**選択**:
- `潜入する`: レジストリ空 OR パーティ未編成 で disabled
- `破棄`: レジストリ空 で disabled
- `新規生成`, `戻る`: 常に有効

**代替案**: 旧条件 (選択なし を含む) のまま

**理由**:
- 新フローでは「選択なし」状態が構造的に存在しない (ボタン列が起点で、必要なら後からリスト焦点移動)
- 旧条件を残すと、「リスト焦点時の選択状態を裏で保持する」という暗黙の状態が発生して混乱源になる
- レジストリ空は `LIST_FOCUS_*` に入る意味がない (選択対象がない) ため、ボタン自体を disabled にするのが論理的

### 決定4: 空レジストリ時は `新規生成` に初期カーソル配置

**選択**: 既存要件 `Initial focus adapts to empty dungeon registry` の主旨は残すが、理由を書き換える:
- 旧: 「リスト起点だと空で操作不能なので新規生成に初期配置」
- 新: 「`潜入する` `破棄` が disabled になるので、最初の有効ボタン `新規生成` にカーソル初期配置」

**代替案**: 一律 `潜入する` に初期配置

**理由**:
- 空状態でのユーザ体験 (Enterを一発押せばダンジョン作成に進める) を維持したい
- 新フローは常にボタン列起点なので、「カーソル位置の初期値」だけが分岐点になり、実装がシンプル

## Risks / Trade-offs

- **[リスク] 既存テストの互換性**: `DungeonEntrance` のフローテストは現行の「リスト起点 → Enter → ボタン」シーケンスを前提にしている可能性が高い → **軽減**: テストを新フローに合わせて書き換える。既存ユニットテストで無効化されるものはコメントではなく削除し、新しいシナリオのテストを同時に追加する
- **[リスク] プレイヤーの認知変化**: 既にプレイ慣れしている人には「リストから始まらない」ことが一時的に違和感 → **軽減**: ヒントテキスト (画面下部) で操作順を明示
- **[リスク] パーティ格子だけ `"▶ "` が inline で残ることへの spec 不整合**: `cursor-menu-ui` の「prefix-string API removed」要件と衝突しないか → **軽減**: inline 2D格子は `CursorMenuRow` の対象外 (選択文字はレイアウト文字列の一部であり、UI部品の prefix API ではない) と `party-formation/spec.md` で明示する
- **[トレードオフ] 一覧の情報密度**: アクションを選ばないと対象を指定できない = 一覧を見てから「やっぱりこれにする」と切り替える手数が増える → 今回は動詞先行のメリット優先で受容

## Migration Plan

1. spec を先に更新 (`dungeon-entrance/spec.md`, `party-formation/spec.md`)
2. `PartyFormation` の待機リストを `CursorMenuRow` ベースに書き換え + 格子の `CURSOR` 定数を差し替え
3. `DungeonEntrance` を状態機械ベースに書き換え (初期フォーカス・遷移・disabled条件)
4. GUT テスト更新 (既存の旧フロー前提のテストを新フローに置換)
5. 手動プレイテスト (街→入口→潜入、街→入口→新規生成、街→入口→破棄→確認、空状態での新規生成フロー)

ロールバック: 個別変更なのでコミット単位で `git revert` 可能。リスクが高いのは `DungeonEntrance` の書き換え部分だが、状態機械を新旧で切り替えるフラグは導入しない (UIは一度に置き換える方がユーザに混乱がない)。

## Open Questions

- ヒントテキスト (操作方法表示) の文言: 現状の画面下部に表示されているキー表を新フローに合わせて更新する必要があるが、文言は実装時に確定する
- `LIST_FOCUS_ENTER` 中にパーティ未編成だった場合の挙動: そもそも `潜入する` が disabled なので問題は起きないが、万一の保険としてリスト focus 中にも再チェックするか要確認 (実装時の判断で良い)
