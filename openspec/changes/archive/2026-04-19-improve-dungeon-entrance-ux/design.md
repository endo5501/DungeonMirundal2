## Context

ダンジョンの入口 (START タイル) は生成時点で `WizMap` に存在するが、描画層 (3D / ミニマップ) と状態遷移層 (帰還/再入場) のどちらも、このタイルを特別扱いしていない。結果として:

- プレイヤーは広いダンジョンで入口を見失う (視認不能)
- `DungeonData.player_state` は `DungeonRegistry` 内で生存し続けるため、全滅・脱出の巻物・緊急脱出の巻物後の再入場時に「最後の立ち位置」から開始してしまう (意図していない位置保存)

これを解決するため、3 層 (3D メッシュ / ミニマップ / 画面遷移) にそれぞれ最小限の修正を入れる。設計上の最大のポイントは「どこで位置リセットするか」。町→ダンジョン入場の 1 点集中 (案 B3) により、複数の帰還経路 (自発・全滅・スクロール) を条件分岐なしで一元処理できる。

## Goals / Non-Goals

**Goals:**
- プレイヤーが 3D ビューとミニマップの両方で入口タイルを視覚的に特定できる
- 全滅・脱出の巻物・緊急脱出の巻物・自発帰還、いずれの経路でもダンジョン再入場時に必ず入口から再開する
- セーブデータからのロード時は保存地点で再開する挙動を維持する
- 既存の Wizardry 風探索フローの味 (「ミニマップは視野限定」) を崩さない

**Non-Goals:**
- ミニマップの視野半径拡大や「全体マップ」トグルの追加
- HUD に入口方向のアローや距離を表示する
- 「脱出の巻物」「緊急脱出の巻物」の効果本体の変更 (町へ帰還するだけの現状挙動は維持)
- 全滅ペナルティ (ゴールド減・装備ロスト・蘇生必要化) の導入
- START タイル以外の「別出口」「隠し階段」等の新規追加
- ダンジョン生成ロジック (`WizMap.generate`) の変更

## Decisions

### Decision 1: 位置リセットは「入場時」に行う (案 B3)

**採用**: `main.gd:_on_enter_dungeon()` で `DungeonData.reset_to_start()` を呼ぶ。

**理由**:
- 帰還経路が 3 種 (自発 / 全滅 / スクロール) に分岐しているが、「入場」は 1 点に集約している。リセットを入場時にすれば、帰還側に条件分岐を足す必要がない。
- ロード経路 (`_load_game` → `_restore_current_screen`) は `_on_enter_dungeon` を通らないため、セーブからの復元を自動的に保護できる。
- 将来「自発帰還を START 以外の地点から行う」等の拡張を入れても、入場時リセットの方針は影響を受けない。

**代替案**:
- **A 案: 帰還時リセット** — `_on_return_to_town` でリセット。自発帰還と強制帰還で分岐が必要 or 強制リセットにしてしまうと「帰還時点で位置が既にリセット」となり、地上画面のセーブに影響する可能性がある。
- **C 案: `DungeonRegistry` 経由** — レジストリにメソッドを生やす。`DungeonData` 自身の状態変更なのに外部 API が窓口になるのは責務が倒置。

### Decision 2: リセットメソッドは `DungeonData` に持たせる

**採用**: `DungeonData.reset_to_start()` を新設。内部で `_find_start(wiz_map)` を再利用し、`player_state = PlayerState.new(start_pos, Direction.NORTH)` に再生成する。

**理由**:
- 位置を保持する責務は `DungeonData` にある。リセットも同じ場所に置くのが凝集性の観点で自然。
- テストが単純 (`DungeonData` 単体テストのみで検証可能)。`main.gd` 側は「メソッドを呼ぶ」だけの薄い結合。
- `DungeonData.create()` と同じ `_find_start()` ロジックを共有できる (重複なし)。

**代替案**:
- **`main.gd` 内で直接書く**: リセット処理が `PlayerState.new(...)` のインライン生成になり、`DungeonData` の内部実装 (`_find_start`) を外から触る形になる。カプセル化崩れ。
- **`DungeonRegistry` にメソッド**: `reg.reset_player_state(index)` のような形。`DungeonData` を取り出して書き換えるだけの薄いラッパになり、`DungeonData` 側に結局ロジックが必要なので増やす意味が薄い。

### Decision 3: 3D 入口の視覚表現は「上り階段メッシュ」

**採用**: `CellMeshBuilder` が `cell.tile == TileType.START` のセルに対して、床の上に 2〜3 段の簡易上り階段形状の追加メッシュを生成する。階段はセルの中央に配置し、高さは `CELL_HEIGHT` を超えない範囲 (例: 0 〜 1.0 ユニット)。

**理由**:
- Wizardry 風ダンジョンで「ここから降りてきた」というメタファが自然。
- 床色変更だけだと真上からでないと視認困難。立体物なら離れた通路からでも影や形状で気付ける。
- メッシュは既存の `Face` 構造体の繰り返しで組める。新規シェーダやテクスチャは不要。

**代替案**:
- **床色変更のみ**: 最小コストだが視認性が弱い。
- **発光ポータル**: 見た目は派手だが、Wizardry 系のトーンに合わず、エミッシブマテリアルの追加コスト有り。

### Decision 4: ミニマップ入口は「床の上に記号を重ね描き」

**採用**: `MinimapRenderer` が `wiz_map.cell.tile == TileType.START` のセルについて、床色を描画した後、その上に 1〜2 ピクセルの記号 (小さな上向き三角 / 四角など) を重ねる。床色自体は変更しない。

**理由**:
- 「床色変更」にすると他のタイル種別 (FLOOR/GOAL) と色空間が混み合う恐れがある。床色はそのままにして「上に載せる」方が明示的。
- 3x3 ピクセルしかないセル床に描くため、記号は 1〜2 ピクセル程度の単純形状に留める。
- プレイヤーマーカー (中央セル) との色衝突を避けるため、プレイヤー色 (緑) とは別の色を選ぶ。

**代替案**:
- **専用床色**: 実装は最も単純だが、色衝突と差分の曖昧さがリスク。
- **点滅**: 注目度は最高だがレンダリングサイクルを変える必要があり、リターンに対してコストが大きい。

### Decision 5: 位置リセットの呼び出しタイミング

**採用**: `main.gd:_on_enter_dungeon(index)` で `registry.get_dungeon(index)` した直後、`DungeonScreen` をセットアップするより前に `reset_to_start()` を呼ぶ。

**理由**:
- `DungeonScreen.setup_from_data(dd, party)` は `dd.player_state` を参照するため、セットアップ前にリセットしなければ初回描画が古い位置になる。

## Risks / Trade-offs

- **Risk**: 既存のセーブデータで `DungeonData.player_state` が入口以外を指している場合、ロードした直後に「入場」するとリセットされてしまう。
  - **Mitigation**: ロード経路は `_restore_current_screen` → `_show_dungeon_screen` を通るため `_on_enter_dungeon` を経由しない。リセットは発生しない。仕様通りの挙動。

- **Risk**: `CellMeshBuilder` に階段メッシュを追加することで、描画コスト (頂点数) が START セル分だけ増える。
  - **Mitigation**: START タイルはダンジョンあたり 1 セル。合計増加は数十頂点オーダで、パフォーマンスへの影響は無視できる。

- **Trade-off**: `CellMeshBuilder.build_faces` のシグネチャに `Cell` 全体を渡す必要がある (既存は `cell` と `grid_pos` を受けているので変更不要)。ただし内部で `cell.tile` を見るようになるため、タイル種別を追加する度に判定が増えていく可能性はある。
  - 将来 FLOOR/START/GOAL 以外のタイル種別が増えた場合は、タイル種別→装飾メッシュのディスパッチテーブル化を検討する (今回はスコープ外)。

- **Trade-off**: ミニマップの重ね描きはピクセル粒度が細かい (3x3) ため、記号の読みやすさに限界がある。
  - プレイ時は「色の違い」として認識できれば十分。遠方の入口をピンポイントで指し示すというより、視野内に入口セルが来た時に気付けるレベルを目指す。

## Migration Plan

- データマイグレーション不要。既存 `DungeonData.to_dict()` / `from_dict()` のスキーマは変更しない。`reset_to_start()` は状態を生成済みの `DungeonData` に対して呼ぶだけで、シリアライズ形式に影響しない。
- 既存のセーブデータは自動で互換動作する。次回「町からダンジョンへ入場」したときに入口へリセットされる (= 本件の意図した挙動)。
