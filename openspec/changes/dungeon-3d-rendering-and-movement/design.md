## Context

project-foundation-and-dungeon-generation で WizMap（迷路生成）、Cell、Direction、EdgeType、TileType などの純粋ロジッククラスが完成済み。これらは全て RefCounted ベースで、Node ツリーに依存しない。

本 change では、この WizMap データを Wizardry 風の一人称3D視点で描画し、プレイヤーがグリッド単位で移動できるようにする。Godot 4.x の 3D レンダリング機能を活用する。

## Goals / Non-Goals

**Goals:**
- WizMap のセルデータを一人称3D視点で描画する（壁・床・天井・ドア）
- プレイヤーの位置・向きに基づいて視野内のセルを正しく描画する
- グリッドベースの移動操作（前進・後退・左右回転）と壁判定を実装する
- ロジック層（RefCounted）と描画層（Node）を分離し、ロジック層はテスト可能にする

**Non-Goals:**
- テクスチャや高度なライティング（単色マテリアルで十分）
- ミニマップやパーティ表示（dungeon-ui で対応）
- 移動アニメーション・スムーズ補間（将来の改善として検討）
- 戦闘エンカウント

## Decisions

### 1. 描画アプローチ: Godot 3D + SubViewport

**選択:** Godot の Node3D / Camera3D / MeshInstance3D を使い、SubViewport 内で3Dシーンを描画する。SubViewport の出力を SubViewportContainer 経由で2D UI に埋め込む。

**理由:**
- Godot の3Dエンジンがパースペクティブ計算・深度処理を自動的に行う
- 将来テクスチャやライティングを追加する際に拡張しやすい
- SubViewport により、ダンジョン3D表示を画面上部、パーティ情報を画面下部に配置する UI レイアウトが自然に実現できる

**代替案:**
- Canvas2D 描画（線画ベース）→ オリジナル Wizardry に近いがエフェクト拡張が困難
- 事前レンダリング画像 → パターン数が膨大、マップサイズに柔軟に対応できない

### 2. アーキテクチャ: ロジック/描画分離

**選択:** ロジック層（RefCounted）と描画層（Node）を明確に分離する。

```
src/dungeon/
  player_state.gd       # RefCounted: プレイヤー位置・向き・移動ロジック
  dungeon_view.gd       # RefCounted: 視野内セル計算ロジック
  cell_mesh_builder.gd  # RefCounted: セルのメッシュデータ生成
src/dungeon_scene/
  dungeon_scene.gd      # Node3D: 3Dシーン構築・更新
  dungeon_screen.gd     # Control: SubViewport + UI レイアウト
tests/dungeon/
  test_player_state.gd  # PlayerState のテスト
  test_dungeon_view.gd  # DungeonView のテスト
  test_cell_mesh_builder.gd  # CellMeshBuilder のテスト
```

**理由:**
- 既存プロジェクトのパターン（RefCounted ベースの純粋ロジック）に合致
- PlayerState と DungeonView は GUT で完全にテスト可能
- 描画層は薄く保ち、ロジック層のデータを受け取って描画するだけにする

### 3. 移動方式: グリッドベース離散移動

**選択:** プレイヤーはセル単位で移動し、90度単位で回転する。連続的な自由移動は行わない。

**理由:**
- Wizardry の操作感に忠実
- WizMap のグリッドデータと1:1で対応し、壁判定がシンプル
- 既存の `WizMap.can_move()` をそのまま活用できる

### 4. 視野計算: 前方4セル + 左右1セル

**選択:** プレイヤーの向いている方向に最大4セル先まで、左右は各1セル幅を描画対象とする。壁で遮られた先は描画しない。

**理由:**
- クラシック Wizardry の視覚的な印象を再現するのに十分な深度
- 描画セル数が限定的（最大約12セル）でパフォーマンス問題が起きない
- 壁による遮蔽判定はシンプルな前方トラバースで実現可能

### 5. メッシュ構築: ImmediateMesh による動的生成

**選択:** 視野が変わるたびに ImmediateMesh で壁・床・天井のジオメトリを再構築する。

**理由:**
- 移動のたびに視野が変わるため、事前に全セルのメッシュを作る必要がない
- ImmediateMesh は頂点データを直接追加でき、シンプルな矩形の壁・床・天井に適している
- 描画対象セル数が少ない（最大12セル程度）ため、毎フレーム再構築のコストは無視できる

**代替案:**
- 全マップ分の MeshInstance3D を事前生成し表示/非表示を切替 → メモリ消費大、初期化重い
- ArrayMesh → ImmediateMesh より柔軟だが、単純な矩形にはオーバースペック

### 6. セルの3D座標系

**選択:** 1セル = 1単位(2.0m)。床は Y=0、天井は Y=2.0。壁は床から天井までの垂直面。カメラはY=1.0（目線の高さ）に配置。

**理由:**
- 整数座標でセル位置と3D位置の対応が明確
- Godot のデフォルトスケールと相性が良い

### 7. ドアの視覚的区別

**選択:** ドアは壁と異なる色（茶色系）のマテリアルで描画する。形状は壁と同じ平面。

**理由:**
- Non-goals で「テクスチャや装飾の凝った表現」は除外されている
- 色の違いだけでプレイヤーはドアと壁を区別できる
- 将来テクスチャを追加する際のベースになる

## Risks / Trade-offs

- **ImmediateMesh のパフォーマンス**: 視野が変わるたびにメッシュ再構築するが、対象セル数が少ない（最大12）ため問題にならない見込み → 問題が出たらメッシュキャッシュを検討
- **SubViewport のオーバーヘッド**: 追加の描画パスが必要 → ダンジョンシーンは軽量なので影響は軽微。問題が出たら直接 Node3D をシーンに配置する方式に切替可能
- **移動アニメーション非対応**: グリッド移動が瞬間的で硬い印象になる → 将来 Tween による補間を追加可能。現時点では機能的な正しさを優先
- **視野計算の精度**: 複雑な部屋形状で遮蔽判定が不正確になる可能性 → 前方直線+左右の単純なアルゴリズムで開始し、必要に応じて改善
